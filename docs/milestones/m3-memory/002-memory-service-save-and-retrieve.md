# Spec 002: Memory service — save after turn, retrieve recent

## Summary

Implement **`MemoryService`** that **persists** a concise **summary** after each completed assistant turn (generated by a **cheap** model call or heuristic truncation of assistant text—**document choice**) and **retrieves** the last **N** chunks (default 10) for a `sessionId` for injection into the orchestrator.

**Plan:** [Phase 3](../../1-mvp.md#phase-3-memory)

## Scope

### In scope

- `suspend fun appendTurnSummary(sessionId: UUID, clientTurnId: UUID, userText: String, assistantText: String)`
- `suspend fun recentChunks(sessionId: UUID, limit: Int = 10): List<MemoryChunk>`
- Transactional write with conversation turn completion (same DB transaction as assistant turn insert if possible)
- **No cross-session** reads

### Out of scope

- Semantic search / embeddings
- User-facing “forget” UI (m5 may add API)

## Dependencies

- **Prior specs:** [m3/001](./001-memory-chunks-schema.md), [m1/001](../m1-backend-skeleton/001-database-migrations-sessions-turns.md)
- **External:** Optional `MEMORY_SUMMARY_MODEL` if using LLM for summaries

## Interfaces & contracts

### Public API

```text
data class MemoryChunk(
  val id: UUID,
  val sessionId: UUID,
  val summary: String,
  val sourceTurnIds: List<UUID>,
  val createdAt: Instant
)

interface MemoryService {
  suspend fun appendTurnSummary(...): Unit
  suspend fun recentChunks(sessionId: UUID, limit: Int = 10): List<MemoryChunk>
}
```

## Behavior

### Acceptance criteria

1. After a turn completes, `memory_chunks` row exists with non-empty `summary`.
2. `recentChunks` for session A never returns rows for session B.
3. Summary generation **failure** does not fail the whole turn (log error; continue)—document.

### Edge cases & errors

- Very long assistant text → summary truncates to **≤ 1024** tokens equivalent—document algorithm
